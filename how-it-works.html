01 — Overview

What does server.py do?

It's the entire backend for the Solana Faucet. A single Flask Blueprint that handles everything — from verifying you're human, to constructing a Solana transaction and sending you 0.001 SOL.

⚡

SOL Transfers

Constructs and signs Solana transactions using solders. Sends 0.001 SOL per claim from the faucet wallet to the requester's address with priority fees.

execute_sol_payout()
✉

Email Verification

Gates claims behind a 6-digit code sent via Gmail SMTP. Codes expire in 5 minutes. Each verified email produces a token used in the claim request.

send-code → verify-code
⚙

Drag-and-Drop Captcha

Custom captcha system — server picks a random crypto name ("Solana", "Bitcoin"...), sends 4 options. User must drag the correct one into a "vault" on the frontend.

api/captcha/generate
⌚

Rate Limiting

5-minute cooldown enforced per wallet address, IP address, and email hash. Prevents the same person from claiming through different vectors.

check_cooldown()
☰

Stats & History

Tracks total claims, unique wallets, SOL distributed. Exposes recent claims with shortened wallet addresses. Faucet balance is cached for 30 seconds.

api/stats · api/recent-claims
⚙

Feature Flags

Four boolean switches at the top of the file control what's enabled: faucet payouts, token CA display, email requirement, and captcha. Exposed via /api/config.

FAUCET_ACTIVE · CA_ACTIVE

03 — API Endpoints

Every route the server handles

All endpoints are mounted under the /solanafaucet URL prefix. Click any row to expand.

GET / Serve the frontend HTML page

Renders the solanafaucet_index.html template. This is the single-page app with both the Modern and Original faucet interfaces baked in.

GET /api/config Feature flags & settings

Returns the current server configuration so the frontend can adapt its UI. If the faucet is disabled, the disabled message is included.

faucet_active bool — is claiming enabled?
ca_active bool — show $FAUCET token info?
email_required bool — gate behind email?
captcha_required bool — require drag captcha?
contract_address string — $FAUCET CA (if active)
sol_per_claim float — SOL per claim (0.001)
cooldown_seconds int — seconds between claims
GET /api/captcha/generate Generate captcha challenge

Picks a random crypto name as the target, adds 3 decoys, shuffles them. Returns a challenge_id and 4 options each with a unique id. The correct answer's ID is stored server-side. Challenges expire after 5 minutes.

challenge_id UUID — unique challenge identifier
instruction "Drag Solana into the vault."
options[] 4 items, each with id + name
POST /api/send-code Send email verification code

Accepts an email address, generates a 6-digit code, sends it via Gmail SMTP. Rate-limited to one code per email/IP per 60 seconds. The email address is hashed (SHA-256) before storage — the plaintext is never persisted.

email string — recipient address
POST /api/verify-code Verify the email code

Checks the submitted code against stored codes for that email hash. If valid and not expired, marks it used and returns a verification_token + email_hash that the frontend passes along with the claim request.

email string — same email used in send-code
code string — 6-digit code from email
POST /api/claim Claim 0.001 SOL

The main endpoint. Protected by the @require_human_token decorator. Validates the wallet address (base58, 32-44 chars, parseable as a Solana pubkey). Checks email verification + captcha if enabled. Enforces cooldown. Then calls execute_sol_payout() to send the SOL on-chain.

wallet_address string — Solana wallet (required)
human_token string — must start with "V-"
verification_token string — from verify-code (if email on)
email_hash string — SHA-256 hash (if email on)
captcha_id string — challenge_id (if captcha on)
captcha_answer string — selected option id (if captcha on)
GET /api/stats Faucet statistics

Returns total claims, total SOL distributed, unique wallet count, and the faucet's current SOL balance. The balance is cached for 30 seconds to avoid hammering the RPC.

GET /api/recent-claims Last 20 claims

Returns the 20 most recent claims in reverse-chronological order. Wallet addresses are truncated to xxxx...xxxx format. Each entry includes amount, tx signature, and a human-readable "time ago" string.

04 — Claim Flow

What happens when you hit "Claim"

The full journey of a claim request through server.py — every check, every gate.

01 Anti-Bot Check

The @require_human_token decorator fires first. It checks that the request body contains a human_token starting with "V-" and that the honeypot field website is empty. Bots that auto-fill hidden fields get rejected with a 403.

02 Faucet Active?

If FAUCET_ACTIVE is False, the request is immediately rejected with a 503 and the disabled message. This is the kill switch.

03 Validate Wallet Address

The wallet string is checked for length (32-44 chars), character set (base58 only), and then parsed through solders.pubkey.Pubkey.from_string() to ensure it's a valid Solana address.

04 Email Verification

If EMAIL_REQUIRED is on, the claim must include a verification_token and email_hash from a prior /api/verify-code call. Missing either returns a 400.

05 Captcha Verification

If CAPTCHA_REQUIRED is on, the server looks up the captcha_id in the captchas JSON. Checks it hasn't expired (5 min), then compares the submitted captcha_answer to the stored correct_id. The used captcha is deleted.

06 Cooldown Check

check_cooldown() scans recent claims in reverse. If any claim within the last 300 seconds matches the wallet, IP, or email hash, it returns a 429 with the remaining time.

07 Execute SOL Payout

execute_sol_payout() connects to the Solana RPC, loads the faucet keypair, checks the balance covers the transfer + fees, builds a VersionedTransaction with a priority fee instruction + transfer instruction, signs it, and broadcasts. Returns the transaction signature.

08 Record & Respond

The successful claim is appended to faucet_claims.json with timestamp, wallet, IP, email hash, amount, and tx signature. The client receives the tx link to Solscan.

05 — Security Layers

How server.py blocks abuse

Five layers of protection, stacked. A request must pass all of them to get SOL.

L1 Honeypot + Human Token
Hidden "website" field traps bots that auto-fill forms. The human_token must start with "V-" — a value set by frontend JS after user interaction.
L2 Email Verification
6-digit code via Gmail SMTP. Email is SHA-256 hashed before storage. 60-second resend cooldown. Code expires in 5 minutes. One code per email per claim.
L3 Drag-and-Drop Captcha
Server generates a challenge with 4 crypto names. User must drag the correct one. Challenge ID + correct answer stored server-side. One-time use, expires in 5 minutes.
L4 Triple Cooldown
300-second cooldown checked against wallet address, client IP (via X-Forwarded-For), and email hash. All three vectors must be clean.
L5 Balance Guard
Before sending, execute_sol_payout() checks the faucet wallet has at least the transfer amount + 10,000 lamports for fees. If not, the claim is rejected before any transaction is built.
06 — Data Storage

Where everything is stored

No database. Just JSON files written atomically (write to .tmp, then os.replace) in the same directory as server.py.

📄
faucet_claims.json
Every successful claim — timestamp, wallet, IP, email hash, tx sig
📧
faucet_email_codes.json
Pending verification codes — hashed email, code, created_at, used flag
🔒
faucet_captchas.json
Active captcha challenges — challenge_id, correct_id, expiry time
🔑
faucet_private_*.py
Secrets file (not in repo) — RPC URL, keypair, wallet, SMTP creds
atomic write pattern
def _write_json(filepath, data): tmp = filepath + ".tmp" with open(tmp, 'w') as f: json.dump(data, f, indent=2) os.replace(tmp, filepath) # atomic
07 — Solana Integration

How the transaction is built

The execute_sol_payout() function constructs a VersionedTransaction (v0) using the solders library, not the older transaction format.

execute_sol_payout() — simplified
# 1. Connect & load keys client = Client(RPC_URL) sender = Keypair.from_base58_string(SENDER_SECRET) # 2. Check balance covers transfer + fees balance = client.get_balance(sender.pubkey()) if balance < lamports + 10_000: return False, "Insufficient balance" # 3. Build instructions instructions = [ set_compute_unit_price(5000), # priority fee transfer(TransferParams( from_pubkey=sender, to_pubkey=receiver, lamports=1_000_000 # 0.001 SOL )) ] # 4. Compile into v0 message & sign msg = MessageV0.try_compile( payer=sender, instructions=instructions, recent_blockhash=client.get_latest_blockhash() ) tx = VersionedTransaction(msg, [sender]) # 5. Broadcast sig = client.send_transaction(tx) return True, sig # solscan.io/tx/{sig}

09 — Architecture Audit

What needs to be fixed

A security and architecture review of server.py. 3 critical issues are exploitable right now. Click any finding to expand.

Critical Email verification token is never validated server.py:579-585 ▶

The verification_token returned by /api/verify-code is a hash of email_hash + code + timestamp + random. But /api/claim only checks that the token is non-empty — it is never looked up or validated against any stored record.

Any attacker can send any 32-character string as verification_token with any email_hash and the server accepts it. Email verification is completely bypassed — you just need non-empty strings for both fields.

Fix
Store the issued token in the email codes DB entry when it's generated. In /api/claim, look up the token in the codes store and confirm it matches the email_hash. Invalidate it after use.
Critical No file locking — double-spend via race condition server.py:620-640 ▶

The claim flow is: read JSON → check cooldown → send SOL → write JSON. With no threading lock, two concurrent requests both read faucet_claims.json, both pass cooldown (seeing no recent claims), and both send SOL. The atomic os.replace prevents file corruption, but not the double-payout.

With a multi-threaded WSGI server (gunicorn default), this is trivially exploitable by sending two requests simultaneously.

Fix
Add a threading.Lock around the entire read-check-write cycle in /api/claim. Alternatively, migrate to SQLite with EXCLUSIVE transactions for atomic reads and writes.
Critical Captcha is replayable on failed claims server.py:593-608 ▶

The captcha entry is only deleted from faucet_captchas.json after a successful claim (line 607). If the claim fails for any reason (cooldown, invalid wallet, insufficient balance), the same challenge_id + correct_id pair remains valid until it expires in 5 minutes.

An attacker solves one captcha, then reuses it for unlimited claim attempts within the expiry window — effectively bypassing the captcha entirely.

Fix
Delete the captcha entry immediately upon first use, regardless of whether the claim succeeds. Move captcha deletion to before the cooldown check.
High Claim recorded after SOL sent — crash loses the record server.py:620-640 ▶

The transaction is broadcast at line 620. The claim is only persisted to faucet_claims.json at line 640 (after re-reading the file, building the record, and writing). If the process crashes between send and write, the SOL is sent but never recorded — no cooldown is set, so the recipient can claim again immediately.

Fix
Pre-write a "pending" claim record before sending SOL. After the transaction succeeds, update it with the tx signature and mark it "confirmed". This is the same two-phase commit pattern used by payment processors.
High X-Forwarded-For is spoofable — rate limit bypass server.py:263-266 ▶

get_client_ip() trusts X-Forwarded-For unconditionally. Any client can send this header with a different IP on every request, getting a fresh rate limit slot each time. The entire IP-based cooldown layer becomes useless.

Fix
If behind a trusted reverse proxy, use Flask's ProxyFix middleware and configure TRUSTED_PROXIES. If running directly, ignore X-Forwarded-For entirely and use request.remote_addr.
High exit(1) in Blueprint import kills the parent process server.py:58-61 ▶

If the secrets file is missing, the Blueprint calls exit(1). In a modular Flask app, this kills the entire parent process — including all other Blueprints. A Blueprint should never call exit(); it should raise an ImportError or RuntimeError and let the parent handle it.

Fix
Replace exit(1) with: raise RuntimeError("faucet_secrets.py not found in " + BASE_DIR)
High check_cooldown breaks early on unsorted entries server.py:289-301 ▶

The loop iterates claims in reverse and breaks at the first entry older than COOLDOWN_SECONDS. This assumes claims are strictly ordered by timestamp. But the race condition above means concurrent writes can produce out-of-order entries. When the loop hits an old entry first, it stops — and skips newer matching claims that should trigger a cooldown.

Fix
Remove the early break. Iterate all claims and check all entries whose timestamp is within the cooldown window, regardless of order. Alternatively, sort claims by timestamp descending before iterating.
Medium require_human_token is trivially defeated server.py:386-396 ▶

The check is: human_token starts with "V-" and the honeypot website field is empty. The frontend generates the token as 'V-' + Date.now() + '-' + random. Any bot that reads the public source code can construct a valid token in one line. The "ANTI-BOT MIDDLEWARE" label overstates what this provides.

Fix
Treat this as a lightweight deterrent only, not a security boundary. For real bot protection, add HTTP-layer rate limiting (Flask-Limiter or nginx limit_req) alongside the existing captcha system.
Medium No attempt limit on /api/verify-code — brute-forceable server.py:516-554 ▶

There are 1,000,000 possible 6-digit codes. The verify endpoint has no per-code attempt limit. At 10 requests/second with no HTTP rate limiting, an attacker can make 3,000 attempts in the 5-minute code expiry window.

Fix
Add a per-email-hash attempt counter. After 5 failed attempts, invalidate the code and require a new one. Also add HTTP-layer rate limiting on this endpoint.
Medium Frontend API paths are relative — break on different mount index.html:1049-1057 ▶

API paths like 'api/config' are relative URLs. They resolve correctly only when the page is served from the Blueprint's exact prefix (/solanafaucet/). If the page is opened from a file path for testing, or the mount point changes, all API calls silently 404.

Fix
Use absolute paths (/solanafaucet/api/config) or inject the base URL from the server via a Jinja template variable at render time.
Medium No structured logging — print() only server.py (throughout) ▶

All logging uses print(). In a gunicorn/systemd deployment, stdout from worker processes mixes together without timestamps or log levels. There is no way to correlate a failed transaction with a specific request, trace a wallet's claim history, or alert on error rates.

Fix
Replace all print() calls with Python's logging module using at minimum "%(asctime)s %(levelname)s %(message)s" formatting. Use logger.error() for failures and logger.info() for successful operations.
Low random module used for code generation, not secrets server.py:269 ▶

generate_code() uses random.choices() which is not cryptographically secure. The Mersenne Twister state can be predicted if enough outputs are observed. For a 6-digit email code this is marginal, but the fix is trivial.

Fix
Replace random.choices(string.digits, k=6) with ''.join(secrets.choice(string.digits) for _ in range(6)).
Low Silent data loss on JSON parse error server.py:237-238 ▶

If faucet_claims.json becomes corrupted (partial write, disk error), _read_json() silently returns an empty list. This means all cooldown checks pass, stats show zero claims, and new writes overwrite the entire history with a fresh list.

Fix
Log the corruption loudly. Do not silently swallow json.JSONDecodeError. Consider keeping a backup of the last known good state before overwriting.
Low JSON flat files won't scale past ~5K claims server.py (all DB helpers) ▶

Every API call reads and deserializes the entire claims file. check_cooldown() is O(N) per claim. api_stats rebuilds a full set of unique wallets on every request. At 10,000+ claims, disk I/O and JSON parsing become the bottleneck. Concurrent writes also mean only one process wins — other writes are silently lost.

Fix
Migrate to SQLite. It handles concurrent reads, row-level operations, and is a drop-in replacement at this scale. The data model maps naturally to a claims table with indexes on wallet, ip, email_hash, and timestamp.
